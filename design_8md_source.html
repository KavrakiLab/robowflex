3 ## Introduction (What is Robowflex?)
4 
5 Robowflex is primarily a wrapper around components from other robotics libraries (in particular, components from _MoveIt!_).
6 The name "Robowflex" is a portmanteau of "Robot" and ["Bowflex"](https://en.wikipedia.org/wiki/Bowflex), as Robowflex was intended just to be better "bench"-marking software (as in ["bench press"](https://en.wikipedia.org/wiki/Bench_press)) than the rest.
7 However, the software needed for doing benchmarking was the same as the software needed for running motion planning in "isolation" from a ROS system while still using ROS components, useful for many other things other than benchmarking.
8 Thus, the project has spun out and become a toolbox for motion planning using real robots that are _MoveIt!_ enabled.
9 
10 The high-level goals of the project are as follows:
11 1. Provide an easy to use interface to complex robotics libraries by brushing the cruft and idiosyncrasies under the rug.
12 2. Provide access to all underlying capabilities of the libraries so users are not hampered by Robowflex in any way.
13 3. Remain untied as possible to the greater ROS system, and support all distros from Indigo onward.
14 
15 This document provides a high-level overview of the design and API of the `robowflex_library`, as well as some of the auxiliary components.
16 Unless otherwise noted, the class or method will be in `robowflex_library`.
17 
18 ## Components
19 
20 There are only a few core components in Robowflex that are necessary for planning.
21 A brief overview of them is provided below.
22 Take a look at the [scripts](scripts.html) to see the components in action.
23 
24 - robowflex::Robot: A representation of a robot.
25 This requires a URDF and SRDF, as well as _MoveIt!_'s joint limits and kinematics plugin information (at least, for _MoveIt!_-based planning).
26 Along with loading information about the robot's geometry, this class provides some helper methods for setting a robot's state.
27 Each robot has a scratch state that can be modified.
28 Under the hood, the robot is represented with _MoveIt!_'s `robot_model::RobotModel`, and the scratch state is a `robot_state::RobotState`.
29 See the detailed description in Robot for more information.
30 
31 - robowflex::Scene: A representation of a planning scene.
32 Requires a Robot to initialize with.
33 Collision objects can be added and moved (as Geometry, see below) and objects can be attached to the robot used for planning (after they have been added to the scene).
34 Under the hood, the scene is represented with _MoveIt!_'s `planning_scene::PlanningScene`.
35 
36 - robowflex::Geometry: Container of both solid primitives and mesh geometry used in planning requests and collision objects in a scene.
37 A few static methods `Geometry::make*` are provided to easily create instances of geometry.
38 The class wraps a `shapes::Shape` and `bodies::Body`, which are used many places in _MoveIt!_.
39 
40 - robowflex::Planner: A motion planner that can compute a plan for a Robot in a Scene.
41 A few default implementations are provided, such as the default OMPL planning pipeline plugin (OMPL::OMPLPipelinePlanner).
42 However, there are many other Planners available, such as the PoolPlanner which enables threaded planning requests, `robowflex_ompl` which gives direct access to the OMPL state space used in _MoveIt!_, and `robowflex_tesseract` provides a planning interface through [tesseract](https://github.com/ros-industrial-consortium/tesseract) and [trajopt](https://github.com/ros-industrial-consortium/trajopt_ros).
43 
44 - robowflex::MotionRequestBuilder: A helper class to help build a motion planning request for a planner.
45 Simplifies the design of complex goal and path constraints, as well as setting start and goal states.
46 
47 - robowflex::Benchmarker: A utility class that simplifies running a set benchmarks.
48 Different requests (A scene, planner, and request) can be added to a list of experiments to run, along with a set of metrics to compute about the results.
49 Then, the benchmark can be run with multiple BenchmarkOutputter classes, which dump results somehow.
50 
51 - robowflex::BenchmarkOutputter: A class that writes benchmarking results to something.
52 A few useful outputters are provided by default, such as the OMPLBenchmarkOutputter, which dumps results into an OMPL benchmarking log format that can be read (after being made into a database) on [Planner Arena](planner_arena.org).

54 ## Connective Tissue
55 
56 One of the core challenges that Robowflex tackles is the issues of input / output in a ROS system (see IO).
57 Robowflex provides a few useful things on this front:
58 - Automatic resolution of file paths and ROS package paths (IO::resolvePath() and IO::resolvePackage()).
59 - Open XML and unprocessed xacro files to strings (IO::loadXMLToString()).
60 - Broad YAML conversion for ROS messages (see [`yaml.h`](yaml_8h_source.html) and `io/yaml.h`) compatible with the output dumped by ROS Python and `rostopic echo`.
61 Many of the components provide methods that serialize / deserialize YAML files.
62 - There are also many useful conversions and transformation-related methods in `tf.h`.
63 - ROS bag file reading / writing (see robowflex::IO::Bag).
64 - HDF5 file reading (see robowflex::IO::HDF5File).
65 - Helpful live visualization in RViz through robowflex::IO::RVIZHelper.
66 Offline visualization can be done with Blender through `robowflex_visualization`.
67 See the [readme](robowflex_visualization/README.html) for more details.
68 
69 Additionally, there are a few implementations of robowflex::Robot for some commonly used robots, such as robowflex::UR5Robot, robowflex::FetchRobot, and robowflex::R2Robot.
