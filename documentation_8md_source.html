1 # Robowflex Documentation (a How-To) {#doc}
2 
3 This document covers how to document code in Robowflex, as well as add additional tutorials such as this document here.
4 
5 ## Source Code Documentation
6 
7 Documentation is automatically generated using [Doxygen](http://www.stack.nl/~dimitri/doxygen/), and is placed in `${CATKIN_DEVEL_PREFIX}/share/robowflex_doc/doc/index.html`.
8 You can see the style of commenting throughout `robowflex_library`.
9 Any additional code added to `robowflex_library` or one of the optional modules should be fully documented.
10 Generally, all documentation is done in the header files, with little to no high-level documentation in the source files.
11 However, inline comments in methods are still encouraged, especially if there is some complex computation occuring.
12 
13 ### Class Documentation
14 
15 A typical class declaration looks as follows:
16 
17 \code{.cpp}
18 // There is extensive class forwarding in the library (declaration of shared pointer types). To have Doxygen
19 // recognize these types, we define them directly above the class as follows:
20 
21 /** \cond IGNORE */
22 ROBOWFLEX_CLASS_FORWARD(Example);
23 /** \endcond */
24 
25 /** \class ExamplePtr
26  \brief A shared pointer wrapper for Example. */
27 
28 /** \class ExampleConstPtr
29  \brief A const shared pointer wrapper for Example. */
30 
31 /** \brief A short description of the class goes here.
32  *
33  * A much longer description of the class and some of its intricacies goes in following paragraphs. We use a
34  * line length of 110 characters total. Occasional, a class has a template parameter which is documented in
35  * this block as well.
36  */
37  class Example
38  {
39  public:
40  /** \brief A short description of a typedef to explain what it is.
41  */
42  typedef std::string String;
43 
44  /** \brief Constructor. A short description.
45  * \param[in] arg Description of what this argument is to the constructor.
46  */
47  Example(int arg);
48 
49  // Occasionally you have many functions in a class and would like to group them up functionally in the
50  // documentation. You can declare groups of functions with the following command, and they will appear
51  // grouped in the documentation:
52 
53  /** \name Subgroup
54  * \{ */
55 
56  /** \brief A function that modifies some input.
57  * \param[in] arg_in An input argument.
58  * \param[out] arg_out An argument that is modified or returned by the function.
59  * \return The return value of the function.
60  */
61  int doesStuff(int arg_in, int &arg_out);
62 
63  /** \brief Sometimes functions have templates!
64  * \param[in] arg An argument.
65  * \tparam T What does this template mean?
66  */
67  template <typename T>
68  void doesStuffTemplate(T arg);
69 
70  // This closes the group.
71  /** \} */
72 
73  private:
74  int value_; ///< Member variables are documented like this.
75  };
76  \endcode

78 ### Other Documentation
79 
80 You should also document new namespaces you create.
81 This only needs to be done once, generally in a top-level header.
82 
83 \code{.cpp}
84 /** \brief Contains all stuff relating to stuff.
85  */
86 namespace stuff
87 {
88 }
89 \endcode
90 
91 This enables documentation of top-level methods in that namespace, which should also be documented similar to class methods.
92 
93 If you have some piece of code you want ignored by Doxygen, simple wrap it with the following:
94 
95 \code{.cpp}
96 /** \cond IGNORE */
97 
98 // Your ignored code goes here.
99 
100 /** \endcond */
101 \endcode

103 ### Python Documentation
104 
105 There are a few Python scripts within the Robowflex code base, primarily within `robowflex_visualization`.
106 There is no consistent documentation for these methods yet, but every method you create should have at the very least a docstring as follows:
107 
108 ```py
109 def method()
110  '''Here is some documentation about method.
111 
112  '''
113  pass
114 ```

116 ## Adding New Tutorials / Pages
117 
118 You can add new pages of high-level documentation like this one in two ways.
119 If you are documentation a package within Robowflex, if a `README.md` exists in the top-level of the project, it is automatically added.
120 See the `add_doc` macro in `robowflex_doc`'s `CMakeLists.txt` for where this is done.
121 
122 Additionally, you can add a new markdown file (ending in a `*.md`) to the `robowflex_doc/doc/markdown` folder.
123 Any markdown file in this directory is also automatically added when generating documentation.
124 
125 ## GitHub Pages
126 
127 The documentation can be read online at [GitHub pages](https://kavrakilab.github.io/robowflex/).
128 To update the documentation on the website, the documentation has to be changed on the `gh-pages` branch of the repository.
129 
130 This is a simple process:
131 1. Generate the documentation for the project with a crisp `catkin build robowflex_doc`
132 2. Change the branch of Robowflex to `gh-pages`
133 3. Make a branch from of `gh-pages`, something like `feature/doc_update`
134 4. Go to `${CATKIN_DEVEL_PREFIX}/.private/robowflex_doc/share/robowflex_doc/doc/html/` for the documentation files.
135 5. Copy the contents of that directory into the top-level folder of Robowflex.
136 6. Commit changes to the branch, and submit a pull request.
